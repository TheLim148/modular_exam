# Шаблоны для модульного экзамена

Набор файлов для быстрого формирования четырёх документов экзамена:

1. Анализ требований (ЛНА + матрица рисков)  
2. Техническое задание (ТЗ)  
3. Спецификация  
4. Оценка качества документации  

Использование рассчитано на выполнение всех модулей **за 2 часа**.

---

## 1. Структура репозитория

README.md
01_requirements.docx
02_TZ_template.docx
03_Spec_template.docx
04_Quality_template.docx
diagrams.md


Каждый документ — отдельный файл с титульником.  
В ТЗ и Спецификацию встроены изображения универсальных диаграмм.

---

## 2. Инструкция: что делать за 2 часа

### Шаг 0 — Узнать тему билета

Используются плейсхолдеры:

<НАЗВАНИЕ СИСТЕМЫ>
<ПРЕДМЕТНАЯ ОБЛАСТЬ>
<ТИПЫ ПОЛЬЗОВАТЕЛЕЙ>
<ОСНОВНЫЕ ФУНКЦИИ>
<КЛЮЧЕВЫЕ ОБЪЕКТЫ ДАННЫХ>

Меняешь только эти блоки.

---

### Шаг 1 — 01_requirements.docx  
(Анализ требований + НПА + матрица рисков)

1. Вставить тему.  
2. В разделе «Нормативные акты» обязательно указать:  
   - Федеральный закон №152-ФЗ «О персональных данных»  
   - Федеральный закон №149-ФЗ «Об информации…»  
   - Один ЛНА (например, «Политика обработки ПДн»).  
3. Краткий анализ требований: 5–7 пунктов по предметной области.  
4. Матрица рисков (в документе): 3–4 риска, вероятность, ущерб, уровень.

Документ 1 готов.

---

### Шаг 2 — 02_TZ_template.docx  
(Техническое задание)

Структура шаблона:

- Описание проекта  
- Назначение и цели  
- Функциональные требования  
- Нефункциональные требования (ссылка на 152-ФЗ)  
- Требования к интерфейсу  
- Технические требования и стек  
- Ссылки на диаграммы  

Что править:

1. Описание предметной области.  
2. Функциональные требования (6–10 пунктов под тему).  
3. Проверить вставленные диаграммы (уже встроены).  
4. Использовать подписи: «см. Рис. 1».

Документ 2 готов.

---

### Шаг 3 — 03_Spec_template.docx  
(Спецификация)

Структура шаблона:

- Постановка задачи  
- Функциональная спецификация  
- Таблицы тестирования  
- Диаграмма архитектуры  
- Блок-схема функции  

Что править:

1. Постановка задачи — заменить описание.  
2. Функциональная спецификация — подставить свои модули.  
3. Таблицы тестирования — заменить названия сущностей.  
4. Диаграммы — уже встроены.  
5. Проверить ссылки на рисунки.

Документ 3 готов.

---

### Шаг 4 — 04_Quality_template.docx  
(Оценка качества)

1. Изменить название системы на титуле.  
2. В таблице выставить баллы по 9 критериям:  
   - Функциональность  
   - Надёжность  
   - Юзабилити  
   - Эффективность  
   - Сопровождаемость  
   - Портативность  
   - Совместимость  
   - Защищённость  
   - Полнота  
3. Итоговый процент считает формула.  
4. Написать вывод (5–7 строк).  
5. При желании — добавить изображение диаграммы оценки.

Документ 4 готов.

---

## 3. Диаграммы (diagrams.md)

В одном файле содержатся:

- Диаграмма прецедентов (Use Case)  
- Пользовательский сценарий (Sequence/BPMN)  
- Архитектурная диаграмма  
- Блок-схема алгоритма  

Все диаграммы уже размещены в ТЗ и Спецификации, код выносится только для удобства.

---

## 4. Что оценивает комиссия

1. **Оформление** — титульники, структура, ссылки на рисунки.  
2. **ТЗ и Спецификация** — соответствие теме, логичность требований.  
3. **Оценка качества** — матрица + расчёты + выводы.  
4. **Нормативные акты и диаграммы** — корректно перечислены и включены.

Каждый блок оценивается от 0 до 2 баллов.

---

## 5. Чего избегать

- Не усложнять диаграммы.  
- Не переписывать структуру шаблонов.  
- Не добавлять ничего вне предметной области.  
- Не тратить время на оформление сверх необходимого.
