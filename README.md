# Шаблоны для модульного экзамена

Данный репозиторий содержит готовые **шаблоны документов** для выполнения модульного экзамена по дисциплине, связанной с оформлением технической документации на программный продукт.

Комплект рассчитан на выполнение **всех четырёх модулей за 2 часа** без переработки структуры документов.

---

## Состав комплекта документов

В репозитории находятся следующие файлы:

- `01_requirements.docx` — анализ требований к информационной системе  
  (локальные нормативные акты, категории данных, матрица рисков);

- `02_TZ_template.docx` — техническое задание (ТЗ);

- `03_Spec_template.docx` — спецификация программного продукта;

- `04_Quality_template.docx` — оценка качества комплекта документации;

- `README.md` — инструкция по использованию шаблонов.

Каждый документ является **отдельным файлом с титульным листом** и уже приведён к формату, ожидаемому экзаменационной комиссией.

---

## Общий принцип работы

Все шаблоны используют **плейсхолдеры**, которые заменяются в соответствии с темой экзаменационного билета.

Основные плейсхолдеры:

<НАЗВАНИЕ СИСТЕМЫ>
<КРАТКОЕ НАЗВАНИЕ>
<ПРЕДМЕТНАЯ ОБЛАСТЬ>
<РОЛИ_1>, <РОЛИ_2>, <АДМИНИСТРАТОР>
<ОСНОВНЫЕ ОБЪЕКТЫ ДАННЫХ>
<ОБЪЕКТ>, <ОБЪЕКТЫ>
<ТИПЫ_ОТЧЁТОВ>
<КЛЮЧЕВАЯ_ОПЕРАЦИЯ>


Изменять структуру документов **не требуется**.  
Достаточно заменить плейсхолдеры и, при необходимости, скорректировать формулировки под конкретную тему.

## Порядок выполнения заданий на экзамене

### Шаг 1. Анализ требований  
**Файл:** `01_requirements.docx`

В документе необходимо:

1. Заменить название системы и предметную область.
2. Заполнить таблицу «Основные объекты информационной системы»
   (2–5 ключевых сущностей достаточно).
3. В разделе нормативных требований **обязательно указать**:
   - Федеральный закон №152-ФЗ «О персональных данных»;
   - Федеральный закон №149-ФЗ «Об информации…».
4. Заполнить матрицу рисков (3–4 строки).
5. Перечислить локальные нормативные акты (2–3 позиции).

Документ не требует большого объёма текста — важна логика и корректность.

---

### Шаг 2. Техническое задание (ТЗ)  
**Файл:** `02_TZ_template.docx`

В ТЗ требуется:

1. Кратко описать предметную область.
2. Заполнить цели и назначение системы.
3. Указать функциональные требования (5–7 пунктов).
4. Нефункциональные требования **обязательно связать с 152-ФЗ**.
5. Проверить, что в тексте есть ссылки на диаграммы
   (например: «см. рисунок А.2»).

Диаграммы прецедентов и пользовательского сценария вставляются в **Приложение А**.

---

### Шаг 3. Спецификация  
**Файл:** `03_Spec_template.docx`

В спецификации необходимо:

1. Заполнить словесную постановку задачи.
2. Указать функциональные модули системы.
3. Адаптировать таблицы тестирования под свою предметную область
   (меняются только названия сущностей).
4. В Приложении А разместить:
   - диаграмму архитектуры;
   - блок-схему алгоритма.

Спецификация должна быть логичным продолжением ТЗ.

---

### Шаг 4. Оценка качества документации  
**Файл:** `04_Quality_template.docx`

Документ оформляется **максимально кратко**:

1. Изменить название системы на титульном листе.
2. Вставить **скриншот Excel-таблицы** с оценкой по 9 показателям:
   - функциональность;
   - надёжность;
   - юзабилити;
   - эффективность;
   - удобство сопровождения;
   - портативность;
   - совместимость;
   - защищённость;
   - полнота.
3. При необходимости — указать итоговую оценку.

Дополнительные описания и развёрнутые выводы не требуются.

---

## Диаграммы

Диаграммы размещаются **не в отдельных файлах**, а **внутри документов**:

- в ТЗ — диаграмма прецедентов и пользовательский сценарий;
- в спецификации — архитектурная диаграмма и блок-схема.

Код диаграмм может храниться отдельно только для удобства,
но на экзамене оцениваются **вставленные изображения**.

---

## Что оценивает комиссия

Оценка проводится по четырём направлениям:

1. Оформление документов (титульные листы, структура).
2. Корректность ТЗ и спецификации.
3. Оценка качества документации.
4. Нормативные акты и диаграммы.

Каждый пункт оценивается от 0 до 2 баллов.

---

## Чего делать не нужно

- Не усложнять диаграммы.
- Не менять структуру шаблонов.
- Не писать избыточные теоретические рассуждения.
- Не добавлять разделы, которых нет в шаблонах.

Данный комплект ориентирован именно на **успешную сдачу экзамена**, а не на промышленную разработку ПО.
